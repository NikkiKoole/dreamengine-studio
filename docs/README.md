# docs/ ‚Äî map & organizing rule

These docs are organized by **genre and durability**, not by topic ‚Äî because the mess
they replaced came from one file trying to be reference *and* status *and* rationale at
once, so part of it was always stale. Each file now answers one kind of question.

> **The rule.** **Status** lives only in `STATUS.md`. **Settled rationale** lives only in
> `decisions/`. **Vision/philosophy** lives only in `VISION.md`. **Exploratory design**
> lives in `design/`. Docs *cross-reference*; they don't *duplicate*. If two docs say the
> same thing, one is wrong ‚Äî fix it at the source and link.

The **engine API reference itself** is not in `docs/` ‚Äî it lives in the code:
`runtime/studio.h` (declarations) and `editor/src/studioDocs.js` (the bilingual one-liners
that drive autocomplete/hover/help). Don't maintain a second copy of the function list in
prose ‚Äî link to those. (This is why the old "~100 functions" counts kept going stale.)

## When you change something

This mirrors **`CLAUDE.md` ‚Üí "Making new things discoverable"**: each new thing goes to its
*home* and gets a one-line pointer from the index ‚Äî `CLAUDE.md` itself stays rules + pointers,
never reference prose.

- New **CLI tool** (`tools/x.js`)? ‚Üí full contract in the file's **header comment**, plus ONE line
  in `CLAUDE.md`'s tools list. Tools are **not** documented in `docs/` ‚Äî the header is the source
  of truth.
- New **design / idea**? ‚Üí a `design/*.md` file, linked from this README's layout tree (a one-line
  blurb). Don't append it to a grown doc; a one-line pointer here is the index.
- Shipped / cut / deferred a feature? ‚Üí update **`STATUS.md`**, then the relevant design note.
- A choice is now *settled* (especially a "no")? ‚Üí write a **`decisions/`** ADR-lite and link to it.
- Proposing new API? ‚Üí **`design/api-notes.md`** (sound ‚Üí **`design/audio-notes.md`**).
- New **audio idea**? ‚Üí the **family index at the top of `design/audio-notes.md`** routes it
  (engines, stations, styles, racks, retrofits ‚Äî one table). Rule of thumb: an audio
  investigation longer than ~a screen gets its own `design/` file + a one-line pointer,
  never a new ¬ß appended to the hub.
- Product direction or a core principle changed? ‚Üí **`VISION.md`**.
- Splitting a section out of a grown doc? **If other files reference its ¬ß-numbers widely,
  keep the numbers verbatim in the new doc + leave a stub at the old spot**
  ([`instrument-engines.md`](design/instrument-engines.md) is the worked example); otherwise just renumber. The kept
  numbering exists *only* to serve live references ‚Äî once those age out, renumbering the
  doc is ordinary cleanup, not a rule violation.
- Moved / split / renamed any doc? ‚Üí **`node tools/lint-docs.js`** ‚Äî verifies every
  relative `.md` link and doc-qualified ¬ß-reference (stub/parent resolutions show as
  soft notes; bare ¬ß-refs are deliberately unchecked). It caught 7 broken links from a
  2026-06 file move on its first run; trust it over grep.
- A doc went stale or started duplicating another? ‚Üí prune it. This README is the contract.
