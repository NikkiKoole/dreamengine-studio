# Editor ↔ cart workflow — the hand-editing gaps

> **Genre: design exploration (scratchpad).** The editor is primarily a
> Nikki + agent collaboration surface, and three workflow gaps bite whenever a
> human hand-edits a cart instead of going through `tools/carts/` + CLI: no
> save-in-place, the two-sprite-sources-of-truth problem (STATUS **open item
> 23**, "the sprite story"), and gallery metadata being write-only-by-CLI and
> read-truncated in the UI. Status of what ships → [`../STATUS.md`](../STATUS.md)
> (items 23 + 26). Related: [`mobile-web-notes.md`](mobile-web-notes.md) (the
> `"mobile":` index.json override is one more hand-maintained metadata field).

---

## Gap 1 — no save-in-place: every save is "Save As"

**Today** (`editor/electron/main.cjs`, `ipcMain.handle('cart:save')` ~line 574):
the save button *always* opens `dialog.showSaveDialog` defaulting to
`mycart.cart.png`. There is no concept of "the cart file currently open" with a
save-back operation — `currentCartFile` (shell.js ~line 886) is tracked, but
only to derive the **publish slug**. Load a cart, tweak two lines, and saving
means re-choosing a path and re-confirming the overwrite, every time.

**Proposal — track the origin, split Save / Save As:**

- When a cart is loaded via dialog, drag-drop, or file path, remember its
  absolute origin path (the `cart:load*` handlers already have it; return it to
  the renderer alongside `name`).
- **Save (Cmd-S)** = re-embed chunks and write to the origin path, no dialog.
  Toast `saved → <name>.cart.png`.
- **Save As** = today's behavior, and it *updates* the origin.
- **Guard: carts loaded from the gallery** (`editor/public/carts/*.cart.png`)
  must NOT save-in-place. Those files are build products of `tools/carts/`
  (regenerated by `make-cart.js`) *and* a shared registry directory the editor
  deliberately never writes (the multi-agent hazard — see the publish-button
  entry in STATUS, thirteenth pass). Gallery-loaded carts have no origin: Save
  falls through to Save As.
- Small print: `cart:save` prefers `build/screenshot.png` as the thumbnail —
  which belongs to whatever ran last. Save-in-place should keep the *existing*
  thumbnail from the origin file unless the user explicitly re-bakes (the
  `--update`/`--run` flow), otherwise a quick text edit can swap a cart's
  carefully baked screenshot for an unrelated frame.

Cheap, contained (one IPC handler + a path variable + a keybinding), no new
formats. This is the highest-value fix of the three.

## Gap 2 — the sprite story (= STATUS open item 23)

Facts and the bite are ledgered in [`../STATUS.md`](../STATUS.md) **open item
23** — short version: editor pixels and a `.cart.js` generator are two sprite
sources of truth that don't know about each other; repainting a generator
cart's sprites ships in *that* build but the next CLI bake silently reverts
them, and plain hand-drawn touch-ups have no path back to `tools/carts/` at
all. This section explores the options; the ledger stays in STATUS.

**Option A — pixels → `.cart.js` exporter (lossless freeze).** Mechanically
trivial: read the 64 slots off the sheet, emit `{ slot: [256 ints] }` (style 3
in the authoring guide), skip all-black slots. An editor button ("export
sprites as .cart.js") or a `make-cart.js --extract-sprites` CLI. The catch: for
generator carts this **freezes a program into data** — a `stamp()`-composed,
`shade()`-lit pipeline becomes an inert array, and the generator's whole point
(parametric variants, the foundry time-lapse) is lost. Right for hand-drawn
carts; destructive for programmatic ones.

**Option B — explicit sprite ownership marker.** One field, e.g.
`spriteSource: "editor" | "generator"` in the `.cart.js` settings (flowing into
`de:settings` like `screenW` does). `make-cart.js` then refuses to rebuild the
sheet for `"editor"` carts (uses the embedded `de:sprites` instead), and the
editor warns before opening the sprite tab on a `"generator"` cart ("these
pixels are baked by `<name>.cart.js` — edits here will be reverted by the next
CLI build"). Doesn't move pixels anywhere; it makes the existing rule
("generator carts get sprite changes in the generator, hand-drawn carts in the
editor — never both") *enforced* instead of remembered.

**Option C — conflict warning only (status quo+).** The publish log already
prints the `⚠ name.cart.js exists` warning (main.cjs ~line 999). Could be
promoted to a visible editor toast when *loading* such a cart. Zero risk, zero
enforcement.

**Lean: B now, A as a tool for hand-drawn carts, C is already half-done.**
B is one field plus two checks and kills the silent-revert failure mode dead;
A is genuinely useful the day a hand-drawn cart needs to enter `tools/carts/`
source of truth (today that path simply doesn't exist). A reverse-engineering
"array → sprite-draw program" exporter is explicitly out of scope (not
feasible, not needed).

## Gap 3 — gallery metadata: can't add, can't edit, can't even read all of it

**Today:**

- Registration in `editor/public/carts/index.json` is a hand/CLI edit (then
  `node tools/lint-carts.js`). This is **deliberate** — the publish button's
  design explicitly never touches `index.json` (shared-registry hazard: parallel
  agents' uncommitted entries; see CLAUDE.md's commit-hazard #2). Any proposal
  here must keep that property.
- Descriptions render clamped to **3 lines** (`shell.css` `.tutorial-desc`,
  `-webkit-line-clamp: 3`) with no affordance to read the rest — and
  descriptions carry the *controls* by convention, so the clamp regularly hides
  the most important line.
- There is no UI to see or edit `kind[]`/`genre`/`homage`/`mobile` at all.

**Proposal, in three independent slices:**

1. **Read it all (cheap, no hazard, do first).** A detail view: clicking a
   gallery card (or an ⓘ) expands/overlays the full description + tags + file
   name. Pure renderer change in `buildTutorialsPanel`. Honestly half of the
   pain is just this.
2. **Edit via a metadata form, write via the existing safe channel.** A small
   "cart info" form (title, description, kind, genre, homage) that does **not**
   write `index.json`. Instead it emits the ready-to-paste JSON entry into the
   build log / clipboard (the `wave editor` "export as code" pattern, applied
   to metadata). Registration stays a deliberate CLI/git act; the editor just
   removes the typing and the format-guessing. Validate with the same
   vocabulary `lint-carts.js` owns (read it from a tiny exported list or just
   duplicate the enum and let lint stay the authority).
3. **(Bigger, only if 1+2 still chafe) — `de:meta` chunk: the cart describes
   itself.** Embed title/description/tags as a fifth zTXt chunk; `index.json`
   becomes *generated* (`lint-carts.js --regen` scans `carts/*.cart.png` and
   writes the registry). Nice properties: a `.cart.png` is then fully
   self-contained (matters for sharing/drag-drop), and publish could register
   carts without the registry race — the regen step is idempotent and
   order-free. Costs: every cart needs a backfill, two sources during the
   transition (exactly the sprite-story disease), and `make-cart.js`,
   `build-site.js`, the editor save path, and lint all touch the chunk. Park it
   as a design direction; don't start it for the convenience win alone.
