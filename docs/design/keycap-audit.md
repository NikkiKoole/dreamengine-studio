# Keycap & touch-retrofit audit

STATUS: EXPLORING (swarm snapshot 2026-06-13) — a worklist, re-run by relaunching the audit; [touch-controls.md](touch-controls.md) has since shipped the shared control band.

> Generated by a 44-agent swarm reading every cart's source (2026-06-13). Each cart was
> classified for the **keycap** pattern (`docs/design/mobile-web-notes.md` §6b — a printed
> key label that should double as a tappable chip) and, more broadly, the right mobile-touch
> fix. This is a **worklist**, not a spec. Re-run by re-launching the audit workflow.

## Prerequisite: build `runtime/keycap.h`

No `keycap()` helper exists yet. Every cart in the keycap section below is blocked on it.
The proposed signature (`mobile-web-notes.md` §6b):

```c
// draws [SPACE] next song — returns true on keyp(KEY_SPACE) OR a tap on the chip
if (keycap(x, y, "SPACE", "next song", KEY_SPACE)) next_song();
```

It's `rect`+`print`+`keyp`+`tapp`, ~30 lines, a cart-land header like `radio.h`. Chips must
respect the ≥16-canvas-px touch floor. Once it exists, the retrofits below are mostly mechanical.

`R` (again) `[` `]` (history) `M` (mute). Best fixed *once in `radio.h`* (a `rad_keycap` row),

## touchControls flag — one-line fixes (114)

Pure `btn()` movement (and/or A/B) games. Add `touchControls: true` to the cart's `.cart.js`
(or call `touch_controls(true)`); the on-screen stick + A/B overlay feeds `btn()`/`btnp()` directly.
No keycaps needed. A few print `Z:`/`X:` labels that would mislead on mobile — worth a wording
pass, but the flag is the unblock.
