Review Session Protocol
Purpose

This document explains how future review sessions should continue, regardless of which AI model or human reviewer is participating.

The goal is not to produce a single final report.

The goal is to gradually build an accurate understanding of DreamEngine while the project is still evolving.

Context

DreamEngine is a living project.

The repository changes daily.

The creator's understanding also changes.

Therefore the review itself is treated as another evolving artifact.

Review notes become part of the project's knowledge.

How to continue a review session

Before starting:

Read Review 0000.
Read all previous review notes in order.
Read any documents or code relevant to the current area.
Continue exploring from the latest understanding rather than starting over.
Working method

The review follows a continuous cycle.

Explore

Read code.

Read documents.

Inspect carts.

Ask questions.

Challenge assumptions.

Allow hypotheses to emerge.

Do not rush toward conclusions.

Discuss

Conversation is part of the review.

Some of the most valuable insights emerge through discussion rather than document reading.

Corrections by the creator are considered valuable evidence.

The reviewer should actively seek calibration rather than attempting to always be correct.

Distill

Whenever a meaningful realization appears:

Create a review note.

A review note captures:

observation
evidence
implications
open questions
confidence

The review note becomes part of the repository.

Continue

Future sessions begin by reading the accumulated review notes.

The understanding should continuously improve.

Earlier conclusions may be refined or replaced.

The history of understanding is considered valuable.

Philosophy

The review is not separate from DreamEngine.

The review is allowed to influence the project.

The project is allowed to influence the review.

New ideas discovered during the review are first-class artifacts.

The purpose is understanding, not judgement.

Important mindset

Do not merely review code.

Attempt to understand:

why the system evolved
how ideas became APIs
how workflows changed
what recurring patterns exist
which ideas are becoming central
which ideas are quietly disappearing

The project should be treated as a living ecosystem rather than a static codebase.

Review style

Prefer curiosity over certainty.

Prefer hypotheses over declarations.

Be willing to revise earlier conclusions.

Look for recurring patterns across:

carts
engine
documentation
tooling
workflows
conversations

The goal is to help uncover ideas that neither the repository nor the creator have fully articulated yet.

Expected outcome

The final report is not the objective.

If the review is successful, the final report should feel almost inevitable because the understanding has already been captured incrementally throughout the review.


ChatGPT review sessions are synchronous.

During a reading session, avoid interrupting with new messages. The review continues only while the model is actively processing. Conversation pauses the reading process.
